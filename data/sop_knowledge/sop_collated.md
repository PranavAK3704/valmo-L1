# Collated SOPs — Valmo L1 Agent

## LOSSES & DEBITS

### Shortage Loss — Scenario 1: Attribution Pending
**Trigger**: Captain received loss AWB info but attribution is still pending
**Process**:
1. Check AWBs are correctly entered
2. Run Metabase loss attribution query — check "Remarks" column for pending status
3. Run Single AWB / Bulk AWB tracking for scan details
4. If still pending → close ticket with pending-attribution message
5. If attribution done → follow Scenario 2
**Escalate to**: L1 handles
**TAT**: 72 hours

### Shortage Loss — Scenario 2: 100% Attribution Marked on Captain
**Trigger**: Captain received loss info, full attribution marked to captain
**Process**:
1. Check AWBs are correctly entered
2. Run Metabase loss attribution query
3. Run AWB tracking queries
4. Check evidence submission sheet (generated by loss team)
5. Check CCTV footage email
6. Validate footage per Shortage SOP
7. Decide:
   - Evidence confirms loss on captain + L1 agrees → close, loss stands
   - Evidence confirms loss on captain + L1 contests → escalate L2 with remarks
   - Captain qualifies for waiver → escalate L2
**Waiver conditions**: Tenure <60 days AND 0 shortage debits AND ≥1 invoice cycle AND shortage pendency=0 AND not GST/COD defaulter
**Escalate to**: L2 if contested or waiver applicable
**TAT**: 72 hours

### Hardstop Loss — Scenario 1: Captain Contesting Loss Marked
**Trigger**: Captain received loss AWB details and is contesting
**Process**:
1. Check AWBs correctly entered
2. Run Metabase hardstop loss attribution query
3. Run AWB tracking (Single/Bulk)
4. Cross-reference Hardstop SOP
5. Decide:
   - Loss correctly marked + no waiver → close with note
   - Loss correctly marked + waiver qualifies → escalate L2
   - Captain processed shipment after receiving hardstop pre-alert → re-route to pre-alert flow
   - L1 contests the loss → escalate L2 with remarks
**Waiver conditions**: Tenure <60 days AND 0 hardstop reversals AND ≥1 invoice cycle AND hardstop pendency=0
**Escalate to**: L2 if waiver or L1 contests
**TAT**: 72 hours

### Hardstop Loss — Scenario 2: Captain Received Pre-Alert Email
**Trigger**: Captain received hardstop pre-alert email, requesting loss not be applied
**Process**:
1. Check AWBs
2. Run Metabase loss attribution + AWB tracking
3. Decide by status:
   - Pre-alert sent + shipment delivered/connected → no loss, close
   - Pre-alert sent + loss NOT marked + captain hasn't delivered/RTO'd → inform 7-day deadline
   - Pre-alert sent + loss IS marked + captain missed deadline → check waiver → if eligible escalate L2, if not close (loss stands)
**Escalate to**: L1 handles (L2 only if waiver eligible)
**TAT**: 72 hours

### Hardstop Loss — Scenario 3: Evidence Shared but Loss Still Attributed
**Trigger**: Captain says evidence was shared but loss still on them
**Process**:
1. Check AWBs
2. Run Metabase loss attribution + AWB tracking
3. Check evidence submission sheet
4. Check CCTV footage
5. Decide:
   - Evidence team confirms loss + L1 agrees + waiver eligible → escalate L2
   - Evidence team confirms loss + L1 agrees + no waiver → close (loss stands)
   - Evidence team confirms loss + L1 contests → escalate L2 with remarks
**Escalate to**: L2 if contested or waiver applicable
**TAT**: 72 hours

## ORDERS & PLANNING

### Drop in Load Volume
**Mandatory inputs**: Hub Code, Current Landing Volume, Duration (1w/2w/3w/1m/2m/3m), Description
**Process**:
1. Review manifestation trend from start date (Query 1: hub manifestation trend)
2. Check hub performance details (Query 2: hub performance with Remarks column)
3. **If any Remarks are blank for a Pin-DC combination → escalate L2**
4. **All other scenarios → close with explanation template**
**TAT**: 12 hours

### Promised Load Not Met
**Mandatory inputs**: Hub Code, Current Manifested Volume, Promised Load, Duration, Description
**Process**: Same as Drop in Load Volume — use same two queries, same escalation rule (blank remarks = escalate)
**TAT**: 12 hours

### Fluctuating Load Volume
**Mandatory inputs**: Hub Code, Current Manifested Volume, Duration, Fluctuation level (daily/weekly), Description
**Process**:
1. Review manifestation trend (Query 1)
2. **If NO fluctuation observed** → close with template
3. **If fluctuation IS observed** → send message to partner AND escalate to L2
**TAT**: 24 hours

## PAYMENTS

### Payment Not Received
**Mandatory inputs**: Payment Cycle, Enbolt ID or Invoice Number
**Process**:
1. Verify mandatory fields present
2. Look up payment status by Hub Code / Invoice No / Enbolt ID
3. Check "Final Status" column
4. Reply with status-specific message:
   - Payment processed
   - Pending sign
   - Hold (GST defaulter)
   - Hold (COD risk threshold)
   - Hold (COD + shipment pendency)
   - Hold (GST > 20L)
   - Hold by ops
   - Onboarding issue
   - F&F Vendor
   - Grocery defaulter
   - Negative balance
   - Payment failed
**Escalate to**: L1 handles
**TAT**: 12 hours

### Invoice Not Received for E-Signing
**Mandatory inputs**: Payment Cycle, Enbolt ID or Invoice Number
**Process**: Same lookup as Payment Not Received. Check e-sign pending status. Reply with status.
**Escalate to**: L1 handles

### Shipment Count Mismatch
**Mandatory inputs**: Payment Cycle, Enbolt ID or Invoice Number
**Process**:
1. Run shipment count reconciliation query
2. Calculate delta (Actual - Invoice) for FWD and RVP
3. Reply with appropriate template (delta=0, delta positive, delta negative)
4. Keep ticket open 24 hours for captain to contest
**Escalate to**: L1 handles
**TAT**: 24 hours

### Wrong Debits — Clarification
**Mandatory inputs**: Payment Cycle, Enbolt ID, CN Number
**Process**:
1. Look up debit reasons per AWB (CN Number)
2. Identify debit type: Bag Shortage, Shipment Shortage, Damage, Dual Scan Mismatch, Hardstop, Intransit, Others, Pilot, Shipment Lost on Field, POD Case
3. Reply with reason for each AWB
4. If partner doesn't respond in 24 hours → auto-dispose
5. If partner requests reversal → follow Wrong Debits Reversal SOP
**Escalate to**: L1 handles
**TAT**: 72 hours

### Wrong Debits — Reversal
**Mandatory inputs**: Payment Cycle, Enbolt ID, CN Number
**Process**:
1. Identify debit reasons per AWB
2. Check if AWBs have had attribution changed/reversals already processed
3. If all AWBs already reversed → close, inform partner
4. If debits still standing → route to L2 Losses team (L2: Wrong Loss Marked → L3: Reason)
**Guardrail**: Only last 3 months eligible. Once payment is processed, no reversal.
**Escalate to**: L2 Losses queue

## CASH HANDOVER

### COD Not Reflecting
**Trigger**: Captain deposited money but not showing in COD pendency
**Process**:
1. Check COD pendency query for hub
2. Check COD transaction history (by hub code or Transaction ID)
3. If pendency = 0 → close with last deposit date + amount
4. If pendency > 0 → investigate further with transaction details
**Escalate to**: L1 handles
